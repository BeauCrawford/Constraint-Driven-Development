# **Constraint-Driven Development - Invariants**

---

## **1. Foundational Identity Invariants**

**CDD_FOUNDATION_CONSTRAINT_PRIMACY**
Constraints hold execution authority over implementation.

**CDD_FOUNDATION_INTENT_PRECEDENCE**
Upstream intent governs all downstream artifacts.

**CDD_FOUNDATION_SEMANTIC_FIRSTNESS**
Meaning must be stabilized before execution is allowed.

**CDD_FOUNDATION_DEVELOPMENT_AS_LOWERING**
Development is a staged semantic lowering process, not direct construction.

**CDD_FOUNDATION_MECHANICAL_REALIZATION**
Code is a realization surface, not a source of truth.

**CDD_FOUNDATION_PROOF_BOUND_AUTHORITY**
No artifact has authority without proof support.

**CDD_FOUNDATION_STRUCTURE_OVER_INTERPRETATION**
Explicit structure outranks interpretive reasoning.

**CDD_FOUNDATION_DECLARATIVE_PRIORITY**
Declarative truth outranks imperative behavior.

**CDD_FOUNDATION_UPSTREAM_DOMINANCE**
Lower layers may not override upstream meaning.

**CDD_FOUNDATION_EXECUTION_AUTHORITY_EMERGENCE**
Execution authority emerges only from semantically closed, proven structure.

**CDD_FOUNDATION_CODE_AS_RESIDUE**
Code is the residual output of satisfied constraints and proof.

**CDD_FOUNDATION_CONFORMANCE_OVER_INVENTION**
Implementation must conform to defined structure, not invent behavior.

**CDD_REQUIREMENT_SCOPE_BOUNDARY**

**CDD_REQUIREMENT_ASSUMPTION_VISIBILITY**
